# handoff sub-capability · planner-version acceptance-checklist generator

> One of the outputs of the AIGD package's `aigd-handoff` (after finalization). Automatically translates the **engineering-version Gherkin acceptance cases + config real values** into a checklist that "a planner/QA holds while playing the game and ticks item by item" (md + xlsx).
> The artifact looks like `<systemDir>/<systemName>-05acceptance-planner.md / .xlsx` (already test-run, passing).

## Positioning
- One canonical acceptance source, multiple audiences: **engineering version** (Gherkin, field references, fed to test automation) ↔ **planner version** (real values substituted in, plain language, fed to manual acceptance).
- Belongs to the handoff package, platform-agnostic (md/xlsx); doesn't touch the tech stack.

## Input (drawn from the manifest cross-layer index)
- The system's engineering-version `-05acceptance` (Gherkin, tagged with R-codes)
- The config tables the system references + LocalizationText (the manifest's「Data Layer / Referenced Shared」column)

## Output (a pair)
- `<system>-05acceptance-planner.md` — for humans to read: a per-feature table, each row = test point / precondition / operation / expected (real values) / measured / pass
- `<system>-05acceptance-planner.xlsx` — for QA to tick: an **info page** (usage / test target / update log / to-confirm / progress-stats formulas) + a **checklist page** (a pass/fail/to-test dropdown, frozen header + filter)

## Recipe (reproducible steps)
1. **Parse config**: use `references/scripts/xlsx_dump.py` to dump the relevant tables (openpyxl errors with `Colors must be aRGB`, the script parses directly via zipfile+xml); `resolve_loc.py` resolves LocalizationText text id → Chinese.
2. **Substitute**: replace the field references in the engineering-version assertions (e.g. `levelTable[50].value`) with real values (e.g. `500`), translated into plain-language operation steps.
3. **Leave a trace**: tag each row with an R-code; the expected value parenthetically carries the「source field」— when a number changes, you only refresh by field, **the logic doesn't move**.
4. **Output the pair**: md (for humans) + xlsx (for QA). The xlsx is generated by directly running `references/scripts/gherkin_to_checklist.py <…-05acceptance.md>` (auto-extracts scenarios → checklist page + info page + pass dropdown + progress formulas COUNTA/COUNTIF/IF, the range already covers all rows); the md human-reading version is laid out separately.

## Re-sync (when config updates, verified)
- Re-dump → **diff row by row** against last time → **change only the affected numbers**, leave the rest; record the changes in the「Update Log」.
- Field-referencing is the root cause: when the numeric tables haven't changed, the checklist numbers stay automatically valid (e.g. one update only added evolution-chain naming, with zero numeric change, and the checklist numbers stayed automatically valid).

## Side-effect value: automatically catches config contradictions
- Substituting real values exposes「design doc ↔ config」inconsistencies → directly produces `[to-confirm]`.
- Example: an evolution group is configured with 3 tiers, but in the quality table the "evolvable count" of that quality = 1 → anything past tier 2 is unreachable.

## Read/write contract (paired with the spine)
- **Read**: the system's cross-layer index in the manifest (rules / acceptance / config / referenced shared).
- **Write**: the two planner-version files; when a contradiction is found, note it in the system's「Recheck / to-confirm」column in the manifest.

## Reference implementation (already graduated into the package)
- **`references/scripts/gherkin_to_checklist.py`** — the portable version: `python gherkin_to_checklist.py <…-05acceptance.md> [out.xlsx]`, **auto-extracts cases from Gherkin scenarios** (no longer hand-listing CASES) → outputs an info page + checklist page xlsx. Verified on several systems.
- The companion `xlsx_dump.py` / `resolve_loc.py` (parse config tables / text id → Chinese).
- **The script only does "extract scenarios → checklist structure"**: substituting the real values of the field references in assertions (`levelTable[50].value`) **still needs the config** (look up via `resolve_loc` / the config table, manually or as a later enhancement); the script keeps the original assertion text and doesn't fabricate numbers.

## Limitations
- On Windows without LibreOffice, the xlsx-formula validation script can't run (it depends on a Unix socket); the formulas are trivial, just open in Excel/WPS and it auto-recomputes.
